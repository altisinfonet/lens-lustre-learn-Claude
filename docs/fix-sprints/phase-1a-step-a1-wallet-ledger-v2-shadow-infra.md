## 1. SQL APPLIED

Single migration. Three new tables + one new function. `REVOKE ALL` on the function. RLS enabled + admin-read SELECT policy on all three tables. Live mutation branch is **stubbed with `RAISE EXCEPTION`** so even if EXECUTE were granted, no balance change is possible until a future explicit step lifts the stub.

## 5. IDEMPOTENCY PROOF (static)

Branch B of the function:
```sql
SELECT * INTO v_existing FROM public.wallet_ledger_idempotency
 WHERE op = p_op AND idempotency_key = p_idempotency_key;
IF FOUND THEN
  INSERT INTO wallet_ledger_audit_log(... result='replay' ...);
  RETURN jsonb_build_object('ok', true, 'replay', true, ...);
END IF;
```

`wallet_ledger_idempotency` enforces `PRIMARY KEY (op, idempotency_key)`, so the live branch (when later enabled) cannot register two rows for the same `(op, key)`. The dry-run branch deliberately does **not** insert into the idempotency table — only the future live branch will, so a dry-run today + live tomorrow on the same key will not collide.

## 6. INVALID DEBIT (OVERDRAFT) PROOF (static)

Branch D:
```sql
v_balance_after := v_balance_before + p_amount;
IF v_balance_after < 0 THEN
```

Negative resulting balance is rejected before any further work.
